The MSDK build system manages the compilation of source code into program binaries and exposes a **Command-Line Interface (CLI)** for setting **Build Configuration Variables**. All supported IDEs interface with this system.

## Core files

Every MSDK project contains two files at its root directory that drive the build:

| File | Purpose |
|------|---------|
| `Makefile` | The core build file. Do **not** edit this directly. It exposes the CLI and provides a default configuration suitable for most projects. |
| `project.mk` | Your project-specific configuration file. Edit this file to customize the build. |

The `Makefile` is shared across all MSDK projects and is managed by Analog Devices. The `project.mk` file is yours to modify — it is the stable, recommended place for all project-level customization.

<Note>
Both files are used by [GNU Make](https://www.gnu.org/software/make/), which is included with the MSDK toolchain.
</Note>

## Default build behavior

By default, the build system **auto-searches** the project's root directory for source (`.c`) and header (`.h`) files. The optional `include/` and `src/` subdirectories are also searched if they exist:

```bash
Root Project Directory
├─ project.mk
├─ Makefile
├─ *.h
├─ *.c
├─ include/       # Optional — header files
│  └─ *.h
└─ src/           # Optional — source files
   └─ *.c
```

Each project also comes pre-configured for a specific target microcontroller and its primary Board Support Package (BSP). This default configuration is suitable for most use cases.

## Running a build

Navigate into a project folder and run `make`:

```bash
make
```

For faster builds using parallel jobs (console output may interleave):

```bash
make -r -j
```

For a serial build with clean, ordered output:

```bash
make -r
```

<Tip>
The build output is placed in the `build/` subdirectory. By default, the output filename uses the lowercase target microcontroller name (for example, `max78000.elf`).
</Tip>

## Cleaning build artifacts

<Steps>
  <Step title="Navigate to the project folder">
    ```bash
    cd path/to/your/project
    ```
  </Step>
  <Step title="Run a clean command">
    **Project clean** — deletes the `build/` folder and its contents:

    ```bash
    make clean
    ```

    **Full clean** — also removes all compiled peripheral driver libraries:

    ```bash
    make distclean
    ```
  </Step>
</Steps>

<Note>
Use `make distclean` when you change the `TARGET` or `BOARD` configuration, or after significant toolchain updates, to ensure a fully clean rebuild.
</Note>

## Flashing a program

<Steps>
  <Step title="Build the project">
    ```bash
    make -r -j
    ```
  </Step>
  <Step title="Connect a debug adapter">
    Connect a debug adapter (such as the MAX32625PICO included with most EVKITs) between the host PC and the evaluation platform. Refer to the board's datasheet and quick-start guide for hardware setup details.
  </Step>
  <Step title="Flash with OpenOCD via Make">
    ```bash
    make flash.openocd
    ```

    This command builds a target that drives an OpenOCD server behind the scenes to flash **and run** the program binary. It is the recommended approach for command-line flashing.

    <Note>
    `make flash.openocd` was added in the [June 2023 MSDK release](https://github.com/analogdevicesinc/msdk/releases/tag/v2023_06). For full OpenOCD documentation, see the [OpenOCD User Manual](https://openocd.org/doc/pdf/openocd.pdf).
    </Note>
  </Step>
</Steps>

To flash and **halt** the target (for use before a debug session), use the OpenOCD command directly:

```bash
openocd -s $MAXIM_PATH/Tools/OpenOCD/scripts \
  -f interface/cmsis-dap.cfg \
  -f target/<target>.cfg \
  -c "program build/<filename>.elf verify; init; reset halt"
```

Replace `<target>` with the OpenOCD config name for your microcontroller (found in `Tools/OpenOCD/scripts/target/`) and `<filename>` with your build output file.

<Warning>
On Windows, use `%MAXIM_PATH%` (Command Prompt) or `$env:MAXIM_PATH` (PowerShell) instead of `$MAXIM_PATH`.
</Warning>

## Debugging on the command line

<Steps>
  <Step title="Flash and halt the target">
    Use the OpenOCD flash-and-hold command shown above to flash the program and leave the target halted.
  </Step>
  <Step title="Open a new terminal and launch GDB">
    ```bash
    arm-none-eabi-gdb --se=build/<filename>.elf
    ```
  </Step>
  <Step title="Connect GDB to the OpenOCD server">
    ```
    target extended-remote localhost:3333
    ```
  </Step>
  <Step title="Reset and halt the target">
    ```
    monitor reset halt
    ```
  </Step>
  <Step title="Set a breakpoint and run">
    ```
    b main
    continue
    ```
  </Step>
</Steps>

<Tip>
Run `help` inside GDB at any time to see available commands. See the [Common GDB Commands](#common-gdb-commands) reference below for the most frequently used ones.
</Tip>

### Common GDB commands

| Command | Short | Description |
|---------|-------|-------------|
| `monitor halt` | | Halt the microcontroller. |
| `monitor reset halt` | | Reset and immediately halt. |
| `monitor max32xxx mass_erase 0` | | Mass erase flash bank 0. |
| `file <filename>` | | Set the program file. |
| `load` | | Flash the current program file. |
| `continue` | `c` | Continue execution. |
| `break <arg>` | `b <arg>` | Set a breakpoint (function name, `file:line`, or address). |
| `print <variable>` | `p` | Print a variable's value (must be in scope). |
| `backtrace` | `bt` | Print the stack frame. |
| `step` | `s` | Execute the next instruction. |
| `next` | `n` | Execute the next line of code. |
| `finish` | `f` | Continue to the end of the current function. |
| `info reg` | | Print ARM register values. |
| `help` | | List available commands. |
| `quit` | `q` | Quit GDB. |

## Precedence hierarchy

Build Configuration Variables can be set in multiple places. When the same variable is set in more than one location, the following order determines which value wins:

**CLI / IDE > `project.mk` > environment variable > default value**

If a value is set both in an IDE and in `project.mk`, the IDE's value takes precedence. To give `project.mk` maximum precedence, use the [`override` directive](https://www.gnu.org/software/make/manual/make.html#Override-Directive).

For more on setting and using build variables, see [Build Configuration Variables](/build-configuration-variables).
